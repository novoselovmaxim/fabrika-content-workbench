## Контекст

Выполняется после Инструкции 3. Это те разделы исходного спека (`spec1_rus-edit_.md`, разделы 4, 5, 6, 7), которые не являются ядром инновации (это Инструкция 3), но нужны, чтобы продукт выглядел и работал как профессиональный инструмент для агентства/команды, а не персональный генератор постов. ## Раздел A — Governance (согласование и история изменений)

### A.1 Backend

Новый файл: `server/src/api/reviewEvents.ts`, зарегистрировать в `index.ts`.

- `PATCH /api/posts/:id/review-status` — тело `{ reviewStatus, actorName }`. Обновляет `post_items.reviewStatus`, `lastReviewedBy`, `lastReviewedAt`. Одновременно создаёт запись в `review_events` с `eventType: "status_change"`, `payload: { from, to }`.
- `GET /api/posts/:id/review-events` — история событий по посту, сортировка по `createdAt`.

В существующем `server/src/api/posts.ts`, в роуте `PATCH /:id` (обновление поста) — если меняются значимые поля (`title`, `goal`, `hook`, `keyMessage`, `cta`, `scheduledDate`, `status`), логировать это как `review_events` с `eventType: "field_change"`, `payload: { field, from, to }`. Не логировать служебные поля типа `updatedAt`, `sortOrder`.

### A.2 Frontend

В `PostCardPage.tsx` (или где сейчас структура вкладок карточки поста):

- Блок "Согласование" — dropdown с `reviewStatus` (none / internal_review / client_review / approved), вызывает `PATCH /review-status`.
- Отдельная вкладка "История изменений" — таймлайн `review_events`, человекочитаемый рендер `payload` (например: "Статус изменён: черновик → на согласовании клиенту").

В `QueuePage.tsx` (доска/очередь постов) — добавить фильтр по `reviewStatus` или отдельные колонки Kanban под каждый статус.

## Раздел B — Compliance (упрощённый MVP, не полный rule engine)

Важное отличие от оригинального спека: не строить сразу отдельный LLM-классификатор и полноценный редактор правил в Settings. Для первой волны:

### B.1 Backend

Новый файл: `server/src/services/compliance.ts`.

```ts
function checkCompliance(text: string, projectId: string): { riskScore: number; riskTags: string[]; violatedRules: string[] }
```

Логика:

1. Regex-проверка по активным `policy_rules` (глобальным + `projectId`-специфичным), где задан `pattern`. Каждое совпадение добавляет тег в `riskTags` и повышает `riskScore` в зависимости от `severity` (`info` +0.1, `warning` +0.3, `block` +0.6).
2. Не делать отдельный LLM-вызов специально для compliance. Вместо этого переиспользовать risk-скоринг, который уже считается моделью внутри explainable AI layer (Инструкция 3, шаг 3.4) — там модель и так возвращает `risk`/`riskTags` в одном вызове с генерацией контента. `checkCompliance` из этого файла — дополнительный детерминированный слой поверх (regex), который срабатывает даже если LLM что-то пропустил, и не зависит от того, вызывается ли generate() с JSON-форматом.
3. Итоговый `riskScore` для `draft_versions` — максимум из LLM-оценки (шаг 3.4) и regex-оценки (`checkCompliance`), а не сумма (чтобы не задваивать риск за одно и то же).

Базовые глобальные правила (засеять в Инструкции 2, шаг 2.4, если ещё не сделано):

- Запрет обещаний гарантированного медицинского/психологического результата (паттерн вроде `\b(гарантирую|100%\s*результат)\b.{0,40}(вылечит|избавит|уберёт)\b` — уточнить паттерны под реальные кейсы, не полагаться на этот пример как финальный).
- Запрет гарантированного финансового результата.
- Флаг избыточно категоричных обещаний ("навсегда", "раз и навсегда", "100% гарантия").

### B.2 Frontend

- В карточке поста, вкладка AI или отдельный блок "Риски публикации" — список сработавших правил с их `severity`, кнопка "Переписать безопаснее" (повторный вызов генерации с явным указанием `violatedRules` в промпте — попросить модель переформулировать проблемные места).
- В Settings — простой список правил (включить/выключить), без визуального конструктора regex на первой волне. Добавление/редактирование правил — через прямое редактирование записи (форма с полями code/description/pattern/severity), не через сложный UI-билдер.

## Раздел C — Analytics → Insights

### C.1 Backend

Новый файл: `server/src/services/insights.ts`.

```ts
function recomputeInsights(projectId: string): void
```

Минимальный набор на первую волну (без ML, обычная агрегация SQL по уже существующим `analytics_snapshots`, `post_items`, `rubrics`):

- `best_formats` — какие `contentTypeId`/рубрики дают лучший средний `metricValue` (по последним N снапшотам).
- `underused_rubrics` — рубрики, у которых мало постов за последние 30/60 дней относительно остальных.
- `journey_coverage` — по каким стадиям Ханта (`funnelId` → `stages`) есть посты, а по каким пусто (актуально, если у проекта настроена активная воронка).

Результат — запись(и) в `analytics_insights` с `insightType` и `payload` (JSON с готовыми для рендера данными: заголовок, описание, числа).

`POST /api/analytics/:projectId/recompute-insights` — вызывает `recomputeInsights`, лучше по кнопке в UI или раз в день по расписанию (если в проекте уже есть какой-то job runner в `server/src/jobs` — сейчас папка пустая, значит на первую волну делать по кнопке, не городить cron).

### C.2 Frontend

- На `Dashboard.tsx` — блок "Рекомендации" (карточки из `analytics_insights`, самые свежие).
- На `AnalyticsPage.tsx` — вкладка "Insights" со списком всех инсайтов с расшифровкой.

## Раздел D — Мультиязычность (задел на будущее, минимальный объём сейчас)

Поля в схеме уже добавлены Инструкцией 2 (`projects.primaryLanguage`/`supportedLanguages`, `draft_versions.language`, `brand_facts.language`/`canonicalFactId`). На этом шаге — только минимально необходимое, чтобы не блокировать будущую французскую версию, но не строить полноценный мультиязычный UI прямо сейчас:

- В Settings — простой раздел "Языки проекта": выбор `primaryLanguage` (select) и `supportedLanguages` (мультиселект). Без сложной логики переключения интерфейса.
- В формах генерации (`GeneratePanel.tsx` и аналоги) — необязательный параметр `language`, по умолчанию равен `project.primaryLanguage`. Передаётся в `generate.ts`, сохраняется в `draft_versions.language`.
- Explicitly не делать сейчас: перевод самого интерфейса приложения (i18n UI-строк), автоматический перевод существующих `brand_facts` на другой язык. Это отдельная большая задача за пределами первой волны.

## Порядок работы внутри Инструкции 4

Разделы A, B, C, D независимы друг от друга по коду (разные таблицы, разные роуты), можно делать в любом порядке или параллельно. Рекомендуемый приоритет по ценности для позиционирования "для профессионалов/агентств": A (governance) → B (compliance) → C (analytics insights) → D (мультиязычность, чисто задел, не срочно).

## Критерий готовности

- Смена `reviewStatus` поста создаёт запись в `review_events`, видна в UI как таймлайн.
- Публикация поста с заведомо рискованной формулировкой (например, с гарантией 100% результата) даёт ненулевой `riskScore` и понятный `riskTags` ещё до публикации.
- Dashboard показывает хотя бы один осмысленный инсайт на тестовом проекте с историей публикаций.
- Смена `primaryLanguage` проекта не ломает существующий функционал (это задел на будущее, а не активная фича прямо сейчас).
