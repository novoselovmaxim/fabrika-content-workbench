# Инструкция 1 — Чистка хардкода из первой итерации (Берег.Микрошаги)

## Контекст

Fabrika Content Workbench изначально писалась как инструмент под один конкретный проект — психологический блог Берег.Микрошаги. Сейчас цель — универсальный инструмент для любого SMM-специалиста и любой ниши. Часть текстов и правил генерации всё ещё жёстко завязана на психологическую нишу. Это нужно убрать до начала работы над схемой и новыми фичами, иначе новый функционал (governance, compliance, brand_facts) будет строиться поверх генератора, который всё равно продолжит навязывать чужой тон.

Работай аккуратно: не переписывай то, что уже универсально. Ниже — точный список того, что трогать нужно, и то, что трогать не нужно.

## Что менять

### 1.1. `server/src/services/aiGateway.ts`

В массиве `defaultTemplates`:

- Шаблон `caption-carousel` (id: `"caption-carousel"`): в `userPrompt` есть пункт формата "Дисклеймер о самопомощи". Убрать эту строку из списка требований к формату полностью. Это специфика психологической ниши, не универсальное правило поста-карусели.

- Шаблон `visual-prompt` (id: `"visual-prompt"`): в `userPrompt` зашита строка "Тёплая приглушённая палитра (землистые тона, оливковый, персиковый)". Убрать эту строку. Вместо неё палитра должна приходить из данных проекта (активный `brandStyle` из `project.brandStyles`, либо `project.style`). Если у проекта нет заданного стиля/палитры — не подставлять никакую палитру по умолчанию, оставить общую формулировку без конкретных цветов (например: "Палитра и стиль — в соответствии с брендом проекта, см. системный промпт").

- Шаблон `carousel-image-prompt` (id: `"carousel-image-prompt"`): та же строка с землистой/оливковой/персиковой палитрой в `userPrompt`. Убрать по той же логике.

Технически: эти шаблоны сейчас статичны (обычный массив объектов, без доступа к контексту проекта на этапе формирования `userPrompt`). Компонент, который вызывает `fillTemplate`, уже добавляет `buildProjectContext` через system prompt в `generate.ts` — поэтому проще всего просто убрать нишевые строки из `userPrompt`, а системный промпт и так получит актуальный контекст проекта (стиль, тон, brandStyles) через существующий механизм в `generate.ts` (`systemPrompt = systemPrompt + context`). Ничего изобретать не нужно, только вычистить статичные строки.

### 1.2. `server/src/api/generate.ts`

В функции `buildWeekPlanPrompt` есть блок ТРЕБОВАНИЯ со строкой:

```
- Тон: бережный, без агрессивной мотивации
```

Убрать эту строку из шаблона функции. Вместо неё:

- Добавить параметр `projectTone` в объект опций `buildWeekPlanPrompt` (аргумент функции).
- В вызове `buildWeekPlanPrompt(...)` внутри роута `POST /week-plan` подтягивать `project.tone` (проект уже загружается через `db.select().from(projects)...` — если сейчас не загружается в этом роуте, добавить запрос).
- В шаблоне: если `projectTone` задан — вставлять строку `- Тон: ${projectTone}`. Если не задан — не вставлять строку "Тон" вообще, не подставлять "бережный" или любой другой нейтральный дефолт.

### 1.3. Мёртвый код — `content/prompts`

В `aiGateway.ts` объявлена переменная:

```ts
const promptsDir = path.resolve(__dirname, "../../../content/prompts");
```

Она нигде дальше не используется. Папки `content/prompts` и `content/templates` в репозитории пустые (только `.gitkeep`). Это остаток от более ранней архитектуры, когда промпты предполагалось хранить файлами на диске. Удалить эту строку из `aiGateway.ts` (или, если в планах вернуться к файловым промптам — оставить, но пометить комментарием `// TODO: not used yet, reserved for future file-based prompts`). Решение — на усмотрение того, кто это исполняет, но по умолчанию: удалить, чтобы не вводить в заблуждение.

## Что НЕ менять (уже универсально, проверено)

Эти файлы/данные уже написаны без привязки к нише — трогать их не нужно, время на это не тратить:

- `server/src/seeds/contentTextures.ts` — это общая типология форматов контента (кейс, цифры, антикейс и т.д.), применимо к любой нише.
- `server/src/seeds/funnels.ts` — это общие типы воронок продаж (VSL, OPT IN, вебинарная и т.д.), общемаркетинговая методология.
- `server/src/services/hantStages.ts` — лестница Ханта, универсальная модель пути клиента.
- `app/src/components/BrandInterview.tsx` — вопросы онбординга уже нейтральны (в примере ниши: "психология, маркетинг, фитнес, кулинария, IT"; тон "бережно" — один из вариантов на выбор, не единственный).
- `app/src/components/GeneratePanel.tsx`, `server/src/api/knowledge.ts`, `server/src/api/chat.ts`, `server/src/api/strategy.ts` — без нишевых констант.

## Критерий готовности (acceptance test)

Создать тестовый проект в базе с нишей, максимально далёкой от психологии (например: "B2B-консалтинг по логистике"), без заполнения `tone`/`style`/`brandStyles`. Прогнать через:

1. `POST /api/generate/week-plan` — сгенерировать план на неделю.
2. `POST /api/generate` с шаблоном `caption-carousel` — сгенерировать caption.
3. `POST /api/generate` с шаблоном `visual-prompt` — сгенерировать промпт для картинки.

Проверить текст ответов на отсутствие: "бережный", "самопомощь", землистых/оливковых/персиковых оттенков, любых психологических формулировок, которые не были явно заданы в данных этого тестового проекта. Если хоть где-то всплывает — значит, не всё найдено на шаге 1, вернуться к grep.
