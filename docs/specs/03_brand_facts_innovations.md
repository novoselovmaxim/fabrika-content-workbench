# Инструкция 3 — Brand Facts и слой инноваций (explainable AI, экстракция фактов вместо чанков)

## Контекст

Это ядро всей первой волны изменений. Выполняется после Инструкции 2 (таблица `brand_facts` и поля `draft_versions` уже должны существовать в схеме).

Идея: сейчас весь контекст проекта (`server/src/services/projectContext.ts`) собирается в один текстовый блок и целиком суётся в system prompt — ниша, миссия, тон, куски файлов базы знаний, куски одобренной стратегии. Это работает, но: (1) не масштабируется при росте базы знаний, (2) не даёт возможности сказать, какой именно факт был использован при генерации конкретного поста, (3) не позволяет отличить проверенный факт от предположения модели.

Вместо разбиения текста на чанки (классический RAG) мы извлекаем из материалов атомарные факты — короткие проверяемые утверждения о бренде с категорией, источником и уровнем доверия. Это даёт explainability "из коробки": у каждого факта есть источник, у каждого использования факта в генерации — явная ссылка.

## Шаг 3.1 — Сервис экстракции фактов

Новый файл: `server/src/services/factExtraction.ts`.

Функция `extractFactsFromKnowledge(projectId: string, knowledgeEntryId?: string)`:

- Если передан `knowledgeEntryId` — обработать только один элемент `projectKnowledge` (используется при загрузке нового файла/заметки).
- Если не передан — обработать все элементы `projectKnowledge` проекта, у которых ещё нет связанных `brand_facts` (искать по `sourceRef = knowledgeEntry.id`), чтобы не дублировать при повторном запуске.

Промпт для LLM (аналогично другим промптам в `onboarding.ts` — строгий JSON, `responseFormat: "json"`):

```
Извлеки из текста атомарные факты о бренде/продукте. Каждый факт — одно короткое проверяемое утверждение (не абзац, не список).

Категории (выбери одну для каждого факта): product, audience, promise, constraint, proof, faq, other.

Текст:
{content}

Верни JSON-массив:
[{ "category": "...", "factText": "...", "confidence": 0.0-1.0 }]

confidence — насколько явно факт следует из текста (1.0 — прямая цитата/утверждение, 0.6-0.8 — логичный вывод, ниже 0.5 — не включай такой факт вообще).

Только JSON, без пояснений.
```

После парсинга — вставить каждый факт в `brand_facts` с `sourceType: "knowledge_file"` (или `"note"`, в зависимости от типа `projectKnowledge.type`), `sourceRef: knowledgeEntry.id`, `validated: 0`.

Дополнительно — извлечение фактов из уже структурированных данных онбординга (это не требует LLM-вызова, просто трансформация уже существующих JSON-полей в атомарные записи):

- Из `project.valueProp` (JSON с `tasks[]`, `problems[]`, `gains[]` и т.д.) — каждый пункт с высоким score (2-3) становится фактом категории `promise` или `product`, `sourceType: "ai_inferred"`, `sourceRef: "value_prop"`, `confidence: score/3`.
- Из `project.audience` (после deep-анализа на 17 пунктов, см. `onboarding.ts` `generate-audience-deep`) — поля `pains`, `objections`, `desiredResult` для каждой группы становятся фактами категории `audience`, `sourceRef: "audience_deep"`.
- Из `savedCompetitors` — поля `weaknesses` конкурентов можно превращать в факты категории `proof` ("в отличие от X, мы..."), но это опционально, не блокирует MVP.

Функция `deriveFactsFromOnboarding(projectId: string)` — вызывается один раз по кнопке в UI онбординга после прохождения шагов value_prop и audience, не автоматически при каждом сохранении.

## Шаг 3.2 — API для brand_facts

Новый файл: `server/src/api/brandFacts.ts`, зарегистрировать роутер в `server/src/index.ts`.

Эндпоинты:

- `GET /api/brand-facts/by-project/:projectId` — список фактов, с фильтрами `?category=`, `?validated=`, `?sourceType=`.
- `POST /api/brand-facts` — создать факт вручную (`sourceType: "manual"`, `confidence: 1`, `validated: 1` по умолчанию для ручных).
- `PATCH /api/brand-facts/:id` — редактирование текста факта, категории, `validated`.
- `DELETE /api/brand-facts/:id`.
- `POST /api/brand-facts/:projectId/extract` — запустить `extractFactsFromKnowledge` для всего проекта.
- `POST /api/brand-facts/:projectId/derive-from-onboarding` — запустить `deriveFactsFromOnboarding`.

## Шаг 3.3 — Отбор релевантных фактов (без векторной базы)

Новая функция в `factExtraction.ts` или отдельно `server/src/services/factSelector.ts`:

```ts
function selectRelevantFacts(projectId: string, opts: {
  category?: string[];
  hantStage?: number;
  language?: string;
  limit?: number;
}): BrandFact[]
```

Логика на первом этапе — простая фильтрация по полям (WHERE category IN (...), validated = 1 приоритетнее validated = 0, сортировка по confidence DESC), без эмбеддингов и векторного поиска. Обоснование: объём фактов на один проект (десятки-сотни) не требует семантического поиска, достаточно категорийной фильтрации.

Если после фильтрации кандидатов остаётся больше, чем `limit` (например, больше 15-20 фактов) — сделать один дополнительный LLM-вызов: отдать модели список кандидатов текстом и попросить выбрать top-N наиболее релевантных для конкретной темы поста, с кратким обоснованием. Это дешевле и надёжнее, чем разворачивать эмбеддинг-инфраструктуру.

Если в будущем объём фактов на проект вырастет настолько, что фильтрация по категориям станет слишком грубой — следующий шаг не векторная БД, а SQLite FTS5 (полнотекстовый индекс, встроен в SQLite, не требует новой инфраструктуры):

```sql
CREATE VIRTUAL TABLE brand_facts_fts USING fts5(fact_text, content='brand_facts', content_rowid='rowid');
```

Это отдельная будущая оптимизация, не часть MVP.

## Шаг 3.4 — Explainable AI layer в генерации

Изменения в `server/src/services/aiGateway.ts` и `server/src/api/generate.ts`.

1. В `generate.ts`, роут `POST /`, перед вызовом `generate()`:
   - Если есть `postItemId` — получить `funnelId`/`rubricId` поста, определить `hantStage` (через связь topic → funnel/rubric, либо явно передавать с фронта, если стадия уже выбрана пользователем в UI поста).
   - Вызвать `selectRelevantFacts(projectId, { hantStage, limit: 10 })`.
   - Получить релевантную `content_texture` (если пользователь выбрал фактуру в UI поста — сейчас это, вероятно, не хранится явно на `post_items`; если поля нет, добавить `textureCode` на `topics` или `post_items` в правках к Инструкции 2, либо просто передавать необязательным параметром с фронта).

2. Модифицировать системный промпт генерации — добавить блок:
   ```
   ФАКТЫ О БРЕНДЕ (используй только то, что применимо к этой теме):
   [{id}] {factText}
   ...

   Верни ответ в формате JSON:
   { "content": "...", "usedFacts": ["id1", "id2"], "risk": 0.0-1.0, "riskTags": ["..."], "explanation": "1-2 предложения — почему выбраны именно эти факты и формулировки" }
   ```

3. После получения ответа — парсить JSON (`extractJSON`, уже есть в `aiGateway.ts`), сохранять `content` в `contentMarkdown` как раньше, а `usedFacts` → `usedBrandFacts` (JSON), `risk` → `riskScore`, `riskTags` → `riskTags`, `explanation` → `explanation` в `draft_versions`.

4. Важно: не все шаблоны генерации должны переходить на JSON-формат сразу (хук, CTA, короткие форматы — избыточно). В первую очередь перевести `caption-post`, `caption-carousel`, `caption-reel`, `brief` — это основной контент поста, где explainability имеет смысл. Для мелких утилитарных генераций (хештеги, варианты хуков) оставить простой текстовый ответ, не усложнять.

## Шаг 3.5 — UI

Во вкладке AI карточки поста (файл, вероятно, `app/src/components/GeneratePanel.tsx` или соседний компонент для вкладки AI — проверить структуру `PostCardPage.tsx`, там должны быть вкладки):

- Блок "На основе чего": список использованных фактов (подтянуть тексты по `usedBrandFacts` id) + фактура + стадия воронки.
- Кнопка "Почему так" — раскрывает `explanation`.

Во вкладке Drafts:

- Колонки/бейджи `riskScore` (цветовая индикация: зелёный <0.3, жёлтый 0.3-0.6, красный >0.6) и `riskTags`.

Новая страница или раздел в существующей `KnowledgeBase.tsx` (или отдельная `BrandFactsPage.tsx`, по аналогии с другими `pages/*.tsx`):

- Таблица фактов проекта с фильтрами по категории, кнопкой "Извлечь факты из базы знаний" (вызывает `/extract`), возможностью вручную поправить текст факта и проставить `validated`.
- Провалидированные вручную факты должны визуально отличаться от `ai_inferred` (например, галочка/бейдж).

## Критерий готовности

- Для тестового проекта с 3-5 файлами базы знаний экстракция даёт разумный список фактов (не мусор, не дублирующиеся формулировки одного и того же).
- Генерация caption для поста с явно заданной `hantStage` показывает в `explanation` осмысленную причину выбора фактов, а не общую отписку модели.
- `riskScore`/`riskTags` заполняются и видны в UI Drafts.
- Отключение фактов (пустая база знаний у нового проекта) не ломает генерацию — промпт просто не содержит блок ФАКТЫ О БРЕНДЕ, работает как раньше.
